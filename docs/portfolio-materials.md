## 4. 핵심 기술 성과

### 4-1. 적격심사 규칙 엔진 (bid_engine.py, 524줄)

**문제**: 조달청지침 #9269의 별표 1~5 전체(6개 평가 테이블)를 코드로 정확히 구현해야 함.

**해결**:
- 추정가격+공사구분 → 테이블 자동 라우팅 (TABLE_1~5 + TABLE_2B)
- A값 7개 사회보험/안전관리비 항목 합산 (이윤·부가세 제외)
- 순공사원가 98% 배제 판정
- 가격점수 산출: `score = max_score - coeff × |90 - ratio×100|`
- 7단계 분석 파이프라인 오케스트레이션

**기술적 난이도**:
- Python 기본 `round()`는 banker's rounding(IEEE 754) → 조달청 규칙과 불일치 → `Decimal` 전용 헬퍼 3종 직접 구현
- 경계값 `[하한, 상한)` 통일 규칙을 전 모듈에 적용
- 기준율 88%→90% 오류를 조달청 원문 3자 대조로 직접 발견·수정

**검증**: 219개 테스트 전수 통과

### 4-2. 최적 투찰 엔진 (optimal_bid.py, 371줄)

**문제**: 복수예비가격 15개 중 4개가 추첨되어 예정가격이 결정되는 구조에서, 어떤 조합이든 최적인 투찰금액을 찾아야 함.

**해결**:
- C(15,4) = **1,365개** 예정가격 시나리오 전수 생성
- 3단계 계층적 스캔: 100K→10K→1K원 (1,000원 정밀도)
- 전 시나리오 기대값 최대화 투찰금액 도출
- Safe / Base / Aggressive 3전략 출력

**Dual-Precision 아키텍처**:
```
Fast scan: float 기반 _score_fast() → 고속 탐색 (546,000회 연산)
Final:     Decimal 기반 calc_price_score() → 최종 검증 (반올림 오차 제거)
```
- `_round_half_up_float`: `int(value * m + 0.5) / m` 트릭으로 Decimal 없이 ROUND_HALF_UP 재현
- 성능과 정확성의 트레이드오프를 2-pass 설계로 해결

**백테스트 결과 (477건)**:
- 1순위 대비: 승 40.0%, 무 57.9%, **패 2.1%**, 평균 +9.12점
- 예측 정확도: 100% (5점 이내), 84.5% (1점 이내)

### 4-3. 하한율 미달 방지 (BC-46)

**문제 발견**: 부모님 건설 회사의 실제 입찰 220건 분석 → **65%(143건)가 하한율 미달로 즉사**. 경쟁사 서비스를 쓰면서도 절반이 탈락하는 구조적 문제.

**해결**: `floor_rate_bid = ceil(A + floor_rate/100 × (max_scenario_est - A))`
- 최악 시나리오(최고 추정가격) 기준으로 하한율을 수학적으로 보장
- 백테스트 94.7% 효과

### 4-4. 5단계 자동 데이터 파이프라인

**구조**: Docker 스케줄러 → 24시간 주기 자동 실행
```
Pipeline 1: 공고 수집 (서버사이드 업종 필터)
Pipeline 2a: 낙찰결과 수집
Pipeline 2b: 실계약 수집 + 누락 공고 자기치유(self-healing) 백필
Pipeline 3: A값 + 복수예비가격 보강
Pipeline 5: 미확정 공고 재확인 (상태 머신 기반)
```

**기술 포인트**:
- **Write/Read 정합성**: 4-State FSM (`pending → checked_missing → confirmed → error`)
- **자기 치유**: Pipeline 2b에서 누락 공고 발견 시 `inqryDiv=2`(번호 기반)로 자동 복구
- **서버사이드 필터**: API 호출량 대폭 절감 (전체 → 업종별)
- **JSON 원본 이중 보관**: Docker 장애 후 DB 전량 소실 경험에서 도출한 정책

### 4-5. 컨설팅 업체 클러스터링 EDA (Phase 2)

**가설**: "대형 컨설팅 업체가 고객 수로 복수예비가격 번호를 통제한다"

**검증 과정**:
1. 몬테카를로 시뮬레이션 10,000회 → 번호 집중도 = RANDOM → **가설 기각**
2. Co-bidding 기반 커뮤니티 탐지 → 클러스터 간 차이 없음 → **기각**
3. 가격 방향 편향 1.656%p 발견 → **부분 지지**
4. 퍼뮤테이션 테스트 버그 발견(LIMIT 200 vs 무제한) → 수정 후 z=122.62, p=0.001 → **클러스터 존재 확인**

**의미**: 가설을 정직하게 기각하고, 버그를 찾아 수정하고, 대안 경로를 찾아가는 데이터 사이언스 프로세스.

> 다른 각도(가격 방향 편향)에서 통계적으로 유의미한 실마리를 발견했습니다.

## 10. 면접 예상 질문 + 답변 재료

### Q. "왜 이 프로젝트를 만들었나요?"
부모님이 건설 회사를 운영하시는데, 기존 입찰 컨설팅 서비스의 추천가격이 자주 틀렸습니다. 실제 데이터 220건을 분석해보니 65%가 하한율 미달로 즉사하고 있었고, 이건 알고리즘이 아니라 구조적 문제였습니다.

### Q. "기술적으로 가장 어려웠던 부분은?"
Decimal 정밀도 문제. Python의 기본 round()는 banker's rounding이라 조달청 규칙과 결과가 다릅니다. 가격점수 1점 차이로 낙찰이 갈리는 영역에서 부동소수점 오차는 치명적이라, Decimal 전용 헬퍼를 만들고 테스트로 검증했습니다.

### Q. "AI 도구를 활용했다면, 본인의 기여는 뭔가요?"
도메인 규칙 해석(조달청지침 원문 3자 대조), 아키텍처 의사결정(시나리오 엔진→순수 수학 전환), 데이터 분석 설계(가설 수립→검증→기각→피벗), 테스트 전략. AI가 코드를 빠르게 써주지만, 무엇을 만들지 결정하는 건 사람의 몫입니다.

### Q. "3주 만에 이걸 다 했다고요?"
AI 도구를 적극 활용했습니다. 중요한 건 속도가 아니라, 매일 데이터로 검증하면서 방향을 수정한 점입니다. TABLE_5 최적화에 3일을 쓴 뒤 고객 데이터를 보고 '참여 0건'임을 발견하고 즉시 피벗한 게 대표적 예입니다.

### Q. "가설이 기각되면 실패 아닌가요?"
데이터 사이언스에서 가설 기각은 실패가 아닙니다. 몬테카를로 10,000회로 '번호 통제' 가설을 기각한 덕분에 잘못된 방향에 투자하지 않았고, 대신 가격 방향 편향이라는 새로운 실마리를 찾았습니다. 정직한 검증이 결국 더 나은 결과를 만듭니다.

### Q. "경쟁사 대비 우위가 있나요?"
경쟁사는 고객 수로 복수예비가격 번호를 통제하여 예정가격을 '결정'합니다. 이는 담합 소지가 있습니다. 우리는 3,200만건 데이터로 패턴을 분석하여 예정가격을 '예측'하는 접근입니다. 담합 리스크 없이 동일한 효과를 달성하는 것이 차별화 전략입니다.
